  - 代码里不需要直接写 LDAP/setfacl/哈希链，但原理必须能讲清楚

### 决策
- MCP 是一个 Agent 五个工具，不是五个独立 Agent（端侧算力跑不动）
- authd 没完成不阻塞——从文档提取接口契约 → 定义 Python interface → 写 mock → RAG/MCP 只依赖接口
- 做事顺序：先摸透 authd 接口契约 → 写 MCP → RAG 和 Agent 在 MCP 之上同时推进
- NAS_RAG 的区分度在三个维度：可度量的检索质量体系（85 tests）、权限集成、全本地部署

### 关键洞察
- 面试官筛的不是功能，是设计——你的 RAG 别人也做，但权限+审计+本地部署三个维度别人没有
- 控制面（authd）不动，数据面（RAG/MCP/Agent）可以随便加——这是 Service Mesh 的思想
- Agent 对写操作只建议不执行——这是面试时关于「边界判断」最有杀伤力的一句话
