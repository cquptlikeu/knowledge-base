# 架构决策记录

## 2026-06-27: RRF 融合(k=60)替代加权求和

- **背景**：多路召回(向量+BM25)的分数在量纲和分布上有本质差异，无法直接相加
- **选项**：A) 加权求和 B) RRF(Reciprocal Rank Fusion) C) 学习权重
- **选择**：RRF(k=60)，免归一化、对量纲不敏感、可解释、无需训练数据
- **代价**：放弃了通过学习调优的可能，但 700 文件规模下 RRF 已足够

## 2026-06-27: GT 归因退化为成员判定

- **背景**：资格矩阵在 GT 派生期已筛定正例集，评估期无需逐分量诊断
- **选项**：A) 评估期按分量归因 B) 派生期筛正例+评估期成员判定
- **选择**：B，AD-5 归因退化为 `ranked_id ∈ frozen_positives ? TP : FP`
- **代价**：无法按分量诊断 FP 原因，但简化了归因逻辑并消除了口径漂移

## 2026-07-08: semantic GT 匹配=子串包含(非分词命中)

- **背景**：判定一个文本文件是否算某 query 的语义正例
- **选项**：A) 子串包含(NFC+casefold, AND) B) 分词后 token 命中
- **选择**：A，确定性强、可手算 GT、不把 jieba 版本漂移带进冻结 GT
- **代价**：对同义词/近义表达无感知，但 GT 必须确定性可复现

## 2026-07-08: time 分量统一取 created_epoch

- **背景**：FileRecord 有 created_epoch/modified_epoch/captured_epoch 三个候选
- **选项**：A) 统一 created_epoch B) photo 优先 captured_epoch
- **选择**：A，GT 派生与检索 FilterSpec range 同字段，零漂移
- **代价**：photo 的拍摄时间在 S1 物化期需令 created_epoch=captured_epoch

## 2026-07-08: 文件名字母数字粘连保留整体(q3 不拆)

- **背景**：`Q3Report_v2.xlsx` 的分词策略直接影响 Q3 文件名查询召回
- **选项**：A) 保留整体 `q3`/`v2` 不拆 B) 强拆字母↔数字
- **选择**：A，与 substr_match 命中目标一致，避免孤立数字 token 稀释信号
- **代价**：无法用单数字 token 召回，但 Q3 类查询意图是整体片段

## 2026-07-08: S1 小样本 dry-run 先于全量构建

- **背景**：S1 需要产出 manifest.jsonl/errors.jsonl，直接全量风险大
- **选项**：A) 直接全量 ~700 文件 B) 先 10-20 文件 dry-run 再扩全量
- **选择**：B，尽早暴露字段缺失、抽取异常与不变量破坏
- **代价**：多一步流程，但返工成本远低于全量后修正
