# 知识库联动指令

> 知识库根路径：`D:\knowledge-base`（以下简称 `$KB`）

## 你必须遵守的规则

本文件是项目联动的唯一入口。无论从 KB 目录还是从项目目录启动，**你必须**在三个阶段执行以下操作：会话启动、开发过程中、会话结束。

### 项目文件结构

每个项目在 `$KB/projects/<project-name>/` 下有四个文件：

```
$KB/projects/<project-name>/
├── context.md       ← 项目元信息（技术栈、目标、当前阻塞，始终短小）
├── sessions.md      ← 会话记录（最近 5 条完整，旧记录按月压缩）
├── decisions.md     ← 架构决策记录
└── roadblocks.md    ← 踩坑记录
```

---

## 一、会话启动（你必须执行）

**以下步骤必须在每次新会话开始时执行，先于所有其他工作。**

1. **读取用户画像**：读取 `$KB/context/about.md` 和 `$KB/context/preferences.md`
2. **读取项目上下文**：读取 `$KB/projects/<project-name>/context.md`（文件始终短小，完整读取）
3. **读取最近会话**：读取 `$KB/projects/<project-name>/sessions.md`，**只读取「最近会话」区域**（从 `## 最近会话` 到 `---` 分割线之间，最多 5 条记录）。不要加载「历史摘要」区域
4. **按需读取决策和踩坑**：如果当前任务涉及历史决策或曾踩过的坑，读取 `$KB/projects/<project-name>/decisions.md` 和 `roadblocks.md`

## 二、开发过程中（你必须执行）

- **技术选型或概念查询时**：先搜索 `$KB/wiki/`（先读 `_index.md` 定位，再定向读取相关页面）。如果搜不到或结果不满足需求，**必须**去联网搜索，不能凭记忆回答
- **遇到值得保存的文章/文档时**：保存到 `$KB/raw/definitely/`（明确有价值）或 `$KB/raw/casually/`（随手存）。先询问用户确认具体位置再保存。会话结束时主动告知用户「raw 中有 N 篇待处理」以便后续采集

## 三、会话结束时（你必须执行）

**会话结束前，执行以下 5 个步骤，不可跳过。**

### 步骤 1：写项目会话记录

**一个会话只写一条记录**——不管会话里做了多少件事、讨论了多少个话题，全部合并到一条记录中。禁止按主题或任务阶段拆分成多条。

0. **去重**：先读取 `$KB/projects/<project-name>/sessions.md` 中「最近会话」区域的第一条记录，确认上次已记录了哪些内容。本次只写新产生的内容，不重复已记录的部分
1. 在 `## 最近会话` 标题**正下方**插入新记录（最新的在最上面）
2. 标题格式：`### YYYY-MM-DD 时段 — 简短主题描述`，时段用 `上午` / `下午` / `晚间` / `深夜`
3. 标题下方必须包含：
   - **做了什么**：具体到文件路径、数量、关键操作
   - **什么决定**：当前会话做的选择及理由
   - **什么还没做完**：阻塞项和下一步计划
4. **5 条上限**：保持「最近会话」区域只有 5 条完整记录。超过时，将最旧的那条移入「历史摘要」顶部
5. **月压缩**：当「历史摘要」区域出现上一个月（非本月）的多条独立记录时，将其合并为一段 `### YYYY-MM 月汇总`（10-20 行），只保留关键决策 + 重要里程碑 + 值得记录的踩坑，然后删除被合并的原始记录

### 步骤 2：更新项目状态

更新 `$KB/projects/<project-name>/context.md` 的「当前状态」部分（已完成、正在进行、已知阻塞），使其反映本次会话后的最新状态。

### 步骤 3：记录决策和踩坑

- 本次会话中做出的架构决策 → 追加到 `$KB/projects/<project-name>/decisions.md`
- 本次会话中遇到的坑 → 追加到 `$KB/projects/<project-name>/roadblocks.md`

### 步骤 4：写 Daily Notes

如有实质进展，追加到 `$KB/daily/YYYY-MM-DD.md`。格式：

```markdown
## 时段 会话 — 简短主题描述
### 做了什么
- ...
### 决策
- ...
### 还未完成
- ...
```

文件已存在则追加（多场会话用 `---` 分割线隔开），不存在则新建。不需要严格每天写——有实质进展才写。

**注意**：Daily Notes 是 KB 全局日志，sessions.md 是单项目日志——不要混为一谈，两份都要写。

### 步骤 5：Git 同步

KB 仓库和项目仓库**都必须**提交并推送：

```bash
# 1. 推送 KB 仓库
cd $KB
git add -A
git commit -m "chore: <项目名> 会话记录更新"
git push

# 2. 推送项目仓库（在项目目录执行）
git add -A
git commit -m "<type>: <简要描述本次改动>"
git push
```

- commit message 格式：`feat:` `fix:` `refactor:` `docs:` `test:` `chore:` `perf:` `ci:`
- 如果 KB 仓库尚未初始化 git，提醒用户先执行 `git init && git remote add origin <私有仓库地址>`，但不要代劳

## 附录：文件模板

以下为四个项目文件的标准格式，创建新文件时按此模板生成。

### context.md

```yaml
---
project: <项目名>
stack: [Next.js, Prisma, PostgreSQL]
status: active | paused | completed
---
## 项目概述
一句话描述项目是做什么的

## 当前状态
- 已完成的功能
- 正在进行的功能
- 已知阻塞

## 最近会话
见 sessions.md（最近 5 条）
```

### sessions.md

```markdown
# 会话记录

## 最近会话（最近 5 条，完整保留）

### 2026-07-08 晚间 — 商品列表分页 + 用户认证
- 做了什么
- 什么决定
- 什么还没做完

### 2026-07-07 下午 — 数据库 Schema 设计
- ...

---

## 历史摘要（按月压缩，10-20 行/月）

### 2026-06 月汇总
- 关键决策：选 cursor-based 分页而非 offset（理由：大表性能）
- 重要里程碑：完成了数据库 schema 设计和基础 CRUD
- 踩坑：Prisma migration 在 CI 中重复执行（已修复，见 roadblocks.md）
```

**读取规则**：每次启动只读「最近会话」区域。「历史摘要」仅在需要回溯时用 Grep 按关键词搜索。

### decisions.md

```markdown
## YYYY-MM-DD: <决策标题>

- **背景**：为什么需要做这个决策
- **选项**：考虑过哪几个方案
- **选择**：选了哪个，为什么
- **代价**：这个选择有什么代价/风险
```

### roadblocks.md

```markdown
## YYYY-MM-DD: <问题描述>

- **现象**：报错信息 / 异常行为
- **原因**：根因是什么
- **修复**：怎么解决的
- **预防**：以后怎么避免
```
