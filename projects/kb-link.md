# 知识库联动指令

> 知识库根路径：`D:\knowledge-base`（以下简称 `$KB`）

## 职责边界

本文件只定义“项目联动”的增量规则。全局知识库规则以 `$KB/CLAUDE.md` 为准，包括：会话启动、raw 采集、wiki 查询、Daily Notes、KB Git 同步。

本文件不得重复定义 `$KB/CLAUDE.md` 已覆盖的流程；需要使用全局流程时，只引用对应章节。项目代码仓库的提交与 push 不归本文件管理，由项目自己的 `CLAUDE.md` 或项目规则决定。

## 项目文件结构

每个已接入项目在 `$KB/projects/<project-name>/` 下有四个文件：

```
$KB/projects/<project-name>/
├── context.md       ← 项目元信息（技术栈、目标、当前阻塞，始终短小）
├── sessions.md      ← 会话记录（最近 5 条完整，旧记录按月压缩）
├── decisions.md     ← 架构决策记录
└── roadblocks.md    ← 踩坑记录
```

---

## 一、项目联动启动

先执行 `$KB/CLAUDE.md` 的“会话启动”规则。

只有当当前任务涉及某个已接入项目时，才读取该项目四件套：

1. **读取项目上下文**：读取 `$KB/projects/<project-name>/context.md`（文件始终短小，完整读取）
2. **读取最近会话**：读取 `$KB/projects/<project-name>/sessions.md` 的「最近会话」区域（从 `## 最近会话` 到 `---` 分割线之间，最多 5 条记录），不要加载「历史摘要」区域
3. **按需读取决策和踩坑**：如果当前任务涉及历史决策或曾踩过的坑，读取 `$KB/projects/<project-name>/decisions.md` 和 `roadblocks.md`

## 二、开发过程中的项目联动

- **技术选型、概念查询、资料检索**：执行 `$KB/CLAUDE.md` 的 Query 规则。先查 KB；如果 KB 搜不到或结果不满足当前项目需求，必须联网搜索，不能凭记忆回答。
- **遇到值得保存的项目相关材料**：执行 `$KB/CLAUDE.md` 的 Ingest 规则。项目联动只负责在会话结束时提醒 raw 中有哪些待处理材料。
- **产生项目级决策或踩坑**：先在当前会话中记下，收尾时写入 `decisions.md` / `roadblocks.md`。

## 三、项目联动收尾

会话结束前，先执行以下项目增量步骤；然后执行 `$KB/CLAUDE.md` 的“会话结束规则”（Daily Notes、KB Git 同步等）。

### 步骤 1：写项目会话记录

**一个项目、一个会话只写一条记录**——不管会话里做了多少件事、讨论了多少个话题，全部合并到一条记录中。禁止按主题或任务阶段拆分成多条。

操作规则：

0. **去重**：先读取 `$KB/projects/<project-name>/sessions.md` 中「最近会话」区域的第一条记录，确认上次已记录了哪些内容。本次只写新产生的内容，不重复已记录的部分。
1. **插入位置**：在 `## 最近会话` 标题正下方插入新记录，最新的在最上面。
2. **标题格式**：`### YYYY-MM-DD 时段 — 简短主题描述`，时段用 `上午` / `下午` / `晚间` / `深夜`。
3. **正文必须包含**：
   - **做了什么**：具体到文件路径、数量、关键操作
   - **什么决定**：当前会话做的选择及理由
   - **什么还没做完**：阻塞项和下一步计划
4. **5 条上限**：保持「最近会话」区域只有 5 条完整记录。超过时，将最旧的那条移入「历史摘要」顶部。
5. **月压缩**：当「历史摘要」区域出现上一个月（非本月）的多条独立记录时，将其合并为一段 `### YYYY-MM 月汇总`（10-20 行），只保留关键决策、重要里程碑和值得记录的踩坑，然后删除被合并的原始记录。

### 步骤 2：更新项目状态

更新 `$KB/projects/<project-name>/context.md` 的「当前状态」部分（已完成、正在进行、已知阻塞），使其反映本次会话后的最新状态。

### 步骤 3：记录决策和踩坑

- 本次会话中做出的架构决策 → 追加到 `$KB/projects/<project-name>/decisions.md`
- 本次会话中遇到的坑 → 追加到 `$KB/projects/<project-name>/roadblocks.md`

### 步骤 4：检查项目记忆是否进入 KB Git 真源

项目记忆文件存在不等于已进入 Git 真源。会话结束时检查 `$KB` 的 `git status`：

- 如果 `projects/<project-name>/` 有 untracked 或 modified：说明项目记忆已写入，但 KB Git 同步尚未完整完成
- 这不等于项目联动没有生效
- 判断项目联动是否生效，优先看 `sessions.md` 是否写入本次记录、`context.md` 是否反映最新状态
- KB 仓库的提交与推送按 `$KB/CLAUDE.md` 的 Git 同步规则执行
- 项目代码仓库的提交、commit、push 不归本文件管理

## 附录：文件模板

以下为四个项目文件的标准格式，创建新文件时按此模板生成。

### context.md

```yaml
---
project: <项目名>
stack: [Next.js, Prisma, PostgreSQL]
status: active | paused | completed
---
## 项目概述
一句话描述项目是做什么的

## 当前状态
- 已完成的功能
- 正在进行的功能
- 已知阻塞

## 最近会话
见 sessions.md（最近 5 条）
```

### sessions.md

```markdown
# 会话记录

## 最近会话（最近 5 条，完整保留）

### 2026-07-08 晚间 — 商品列表分页 + 用户认证
- 做了什么
- 什么决定
- 什么还没做完

### 2026-07-07 下午 — 数据库 Schema 设计
- ...

---

## 历史摘要（按月压缩，10-20 行/月）

### 2026-06 月汇总
- 关键决策：选 cursor-based 分页而非 offset（理由：大表性能）
- 重要里程碑：完成了数据库 schema 设计和基础 CRUD
- 踩坑：Prisma migration 在 CI 中重复执行（已修复，见 roadblocks.md）
```

**读取规则**：每次启动只读「最近会话」区域。「历史摘要」仅在需要回溯时用 Grep 按关键词搜索。

### decisions.md

```markdown
## YYYY-MM-DD: <决策标题>

- **背景**：为什么需要做这个决策
- **选项**：考虑过哪几个方案
- **选择**：选了哪个，为什么
- **代价**：这个选择有什么代价/风险
```

### roadblocks.md

```markdown
## YYYY-MM-DD: <问题描述>

- **现象**：报错信息 / 异常行为
- **原因**：根因是什么
- **修复**：怎么解决的
- **预防**：以后怎么避免
```
