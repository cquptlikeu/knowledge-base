一、先说结论

AI Coding 工具普及之后,我筛简历和面试的感受,和很多人想象的相反——不是"候选人变强了所以好筛",而是"大家看着都变强了,但真正能筛出来的难度反而上升了"。

原因很简单:AI 把"实现能力"的门槛拉平了,但实现能力从来不是面试真正在筛的东西。

过去,会不会写代码、能不能把一个需求跑通,本身就是一道分水岭。现在这道分水岭被 AI 抹掉了,简历上几乎都摆着完整的 Agent demo、多智能体协同、漂亮的架构图。但面试要找的是"能不能在真实业务里把事做成的人",这个能力 AI 替不了,且更难在一次面试里看出来。

二、为什么"会用AI"已经不构成差异化

这两年我筛简历最直观的感受是:Agent / LangGraph / MCP / 多智能体这些词,从少数人的亮点,变成了几乎人人都在写的标配。工具普及的速度,远快于理解深度的积累。

这就导致一个尴尬:当一个技能人人都会,它就不再是加分项,而是默认项。默认项的意思是——没有它你会被筛掉,有它你也不会因此被选中。

很多候选人没意识到这个变化,还在把"我用 LangGraph 搭了个多 Agent 系统""我做了个能自主调用工具的 Agent"当作核心卖点。这些在两年前是亮点,现在只是入场券。真正的问题在于:你搭的那套东西,在面试官追问之前,你自己想清楚过吗?

三、面试里一问就分的几个点

不是具体的技术题,而是几类问题。每类我都举一个常见的追问场景。

1. 关于"为什么"——为什么拆成多个子 Agent,而不是一个干完?

"你简历上这个 Agent 项目,为什么拆成三个子 Agent,而不是用一个 Agent 加几个工具调用搞定?"

最常见的回答是"多 Agent 协同效率更高""职责分离更清晰"。这个回答等于没回答——因为下一个问题就是"效率高在哪里?你怎么验证的?职责分离的边界是按什么划的?"

不少候选人到这一步就卡了,因为这个拆分其实是跟着某个开源项目或教程照搬的,自己没想过边界在哪里。

能答上来的人,通常会说出一两个具体的取舍:比如这个 Agent 里规划链路和执行链路用同一套上下文会导致 token 太长且互相干扰,所以拆开;或者工具调用环节容易失败,单独拆成子 Agent 便于隔离重试,不影响主流程已经积累的推理。这些细节背不出来,因为它们只在你真的跑过、卡过、调过之后才会留下。

"为什么"是最难造假的维度。AI 能告诉你多 Agent 的好处是什么,但 AI 不能告诉你"你当时为什么这么拆",因为这个"当时"只存在于你自己的经历里。

1. 关于边界——Agent 什么情况下不该自主决策?

"你这个 Agent,什么情况下它不应该自己做决定,必须回到人工?"

这道题筛掉的人比想象中多。不少候选人会硬答一通"加了置信度阈值""做了 fallback 机制",完全没意识到我在问的是"什么时候根本不该让 Agent 决策"。

能答上来的人,通常会清楚地说出一类场景:比如涉及不可逆操作(下单、删除、转账、发邮件对外),Agent 不应该自主执行,必须人工确认;比如低确定性判断(需要业务规则或领域知识),Agent 可以提建议但不能拍板;比如边界模糊的任务,Agent 不知道自己不知道,这时候要靠规则把决策权拉回人工。

会做和知道什么时候不该做,是两个完全不同的能力层级。前者 AI 能帮你,后者不能。而这个"什么时候不该",恰恰是生产环境和 demo 之间最大的鸿沟——demo 里 Agent 永远在自主发挥,生产里你必须想清楚它发挥的边界在哪。

1. 关于失败——Agent 跑飞了,你是怎么定位和兜底的?

"你这个 Agent 项目,跑飞过吗?比如死循环、调错工具、上下文撑爆。出过这种情况吗,当时怎么处理的?"

这道题筛的不是你有没有踩坑(踩过坑是常态),而是你有没有真正理解那个坑的底层成因。

最常见的回答是"重启就好""加了个超时"。这种回答说明候选人把坑当成了噪音,只想赶紧翻篇。但这个问题其实是一个分水岭——它的真实答案,藏在 Agent 的执行环境里,而这恰好是大多数候选人简历里只字不提的部分。

能答上来的人,会讲清楚失败发生在哪一层:

是 Agent 的推理循环失控(比如它反复调用同一个工具,或者陷入了自我对话的死循环)?这是逻辑层的问题,通常靠给循环加状态约束、限制单轮工具调用次数、或者引入一个"critic"节点做监督来解决。

还是 Agent 执行动作的环境出了问题?比如它生成的代码跑在一个不够隔离的沙箱里,某个异常操作把整个 runner 搞挂了;或者多个 Agent 并发跑的时候,执行环境之间互相污染,一个 Agent 的产物覆盖了另一个的中间状态;或者沙箱本身没有资源限制,一个失控的 Agent 把内存吃满,拖垮了同一台机器上所有正常任务。

这一层一旦讲下去,差距就出来了。我接触的不少候选人能讲到第一层(推理循环),讲到第二层(执行环境的隔离、并发、资源回收)就开始含糊。但真正在生产里跑过 Agent 的人会告诉你,Agent 跑飞往往出事的地方,不是推理,而是执行环境——因为推理层的失败你能从日志看出来,执行环境层的失败往往是隐性的、累积的、甚至会连锁影响其他任务。

这也是为什么我现在看简历会特别注意一点:候选人会不会在项目里体现自己对执行环境、沙箱、并发调度的理解。哪怕只是简单一句"每个 Agent 任务跑在独立容器里、超时自动回收",也比"我用了多 Agent 协同架构"信息量大得多。前者说明你碰过真实问题,后者只说明你搭过框架。

坑的深度,和理解的深度,是正相关的。而 Agent 方向最深的坑,往往都埋在执行环境那一层。

四、AI 时代,面试在筛什么

把上面三类问题合起来看,你会发现一个共同点:它们都在问"你自己的判断",而不是"你知道什么"。

这其实不是 AI 时代才有的变化。好的面试一直都在筛判断力,只是过去判断力和实现能力高度绑定——能写出代码的人,通常也想清楚了问题。现在这两者解耦了,AI 替你写代码,但替不了你判断。

所以现在面试真正在筛的,是三层东西:

需求判断:能不能分清"这个需求里什么是真问题,什么是噪音"

方案判断:能不能在多个可行方案里,选出适合当前场景的那个,并说清楚为什么

边界判断:能不能说清楚"这个方案在什么情况下会失效,失效了怎么办"

这三层,都需要真实业务的磨打。不是项目数量能堆出来的,也不是 AI 能帮你提前准备好的。

五、给候选人的几句实话

1. 简历上的项目,少而深,比多而全值钱得多。一个能扛住深度追问的项目,胜过一堆经不起一问的 demo。
2. "我用 AI 做了 X" 这个句式,在简历里已经没有信息量了。换成"我判断 X 不该用 AI 做,因为……",反而更能引起注意。
3. 面试前,对每个项目问自己三个问题:为什么这么设计?踩过最深的坑是什么?如果重做会怎么改?这三个问题答得越具体,你越不怕追问。
4. 真实实习经历的价值,今年比往年都高。因为只有真实业务才会逼你做取舍。如果没有实习,把一个开源项目贡献做到能讲清细节的程度,也好过一堆同质化的模板项目。
5. 写 Agent 项目的时候,多花一句话讲执行环境——它跑在哪、怎么隔离、失败怎么兜底。这一句话在招聘端的信息密度,远大于你又用了哪个新框架。
6. 最后一句不太好听的:AI 能帮你把代码写出来,但面试官想看的是你这个人能不能扛事——代码只是你扛事的载体,不是扛事本身。如果面试里你只剩"我让 AI 写了什么",那面试官能看到的,就只有 AI,没有你
