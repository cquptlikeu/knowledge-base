# SMB vs NFS vs WebDAV vs HTTP API（四种文件协议）

## 对比维度

| 维度 | SMB | NFS | WebDAV | HTTP API |
|------|-----|-----|--------|----------|
| 目标终端 | Windows | Linux/Unix | macOS/跨平台 | 移动端/脚本 |
| 协议层级 | 应用层 | 内核层 | 应用层（HTTP） | 应用层（HTTP） |
| 认证方式 | NTLM（ldapsam） | UID/GID + Kerberos | HTTP Basic → authd | JWT Bearer |
| 权限执行 | POSIX ACL（内核） | POSIX ACL（内核） | POSIX ACL（内核） | POSIX ACL（内核） |
| 性能 | 中 | 高（内核级） | 中低（HTTP 开销） | 中 |
| 防火墙友好 | 低（445端口） | 低（2049端口） | 高（HTTP 端口） | 高（HTTP 端口） |
| 操作系统挂载 | ✅ Windows原生 | ✅ Linux原生 | ✅ macOS原生 | ❌ 不适合 |
| 程序化访问 | ❌ | ❌ | 中（Rclone） | ✅ RESTful JSON |
| 审计集成 | 弱 | 弱 | 弱 | ✅ 深度集成（authd） |
| 移动端友好 | ❌ | ❌ | 中 | ✅ |

## 为什么不能只用一种

> 企业与家庭环境中终端设备的操作系统高度异构。单一协议无法覆盖全部终端，多协议并存是保障用户体验的必要选择。

没有任何单一协议能同时满足：Windows 原生挂载（需 SMB）、Linux 高性能挂载（需 NFS）、macOS 跨平台协作（需 WebDAV）、移动 App 结构化访问（需 HTTP API）。四种协议互补而非竞争。
