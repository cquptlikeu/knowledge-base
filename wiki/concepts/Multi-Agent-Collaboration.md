# Multi-Agent Collaboration（多 Agent 协作）

多个 AI Agent 通过分工协作共同完成一个复杂任务——每个 Agent 负责一个子任务，Agent 之间通过通信协议和任务分配逻辑协调工作。与单一 Agent 加多个工具调用的架构不同，多 Agent 系统强调「Agent 之间的沟通与协商」。

## 核心思想

把复杂任务拆成多个子任务，交给不同「角色」的 Agent 分别处理——分工带来的好处是每个 Agent 的上下文更聚焦、职责边界更清晰、失败隔离更好。但前提是**拆分是有理由的**，不是为拆而拆。

## 什么时候用

- 单一 Agent 的上下文太长，推理质量和效率都下降
- 子任务之间天然独立，并行执行收益明显
- 不同子任务需要不同的工具集和专业知识配置
- 某个环节容易失败，单独拆出来可以隔离重试

## 什么时候不要用

- 简单确定性任务——一个 Agent 一个工具调用就够了
- 拆分的理由只是「职责分离更清晰」——这经不起面试追问
- 子任务之间有强依赖必须串行——拆分不会带来并行收益

## 必答的面试追问

> 「为什么拆成三个子 Agent，而不是用一个 Agent 加几个工具调用搞定？」

不能只说「职责分离更清晰」或「多 Agent 协同效率更高」。真正有说服力的回答都是有具体取舍的：
- 「规划链路和执行链路用同一套上下文会导致 token 太长且互相干扰，所以拆开」
- 「工具调用环节容易失败，单独拆成子 Agent 便于隔离重试，不影响主流程已积累的推理」

## 两种编排范式

| 范式 | 代表框架 | 编排方式 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **图结构编排** | [[entities/LangGraph]] | 显式定义节点（Agent）+ 边（数据流/控制流），包括循环/分支/条件路由 | 可控、可预测、易于调试 | 需要提前设计流程 |
| **对话式编排** | [[entities/AutoGen]] | Agent 之间通过自然语言对话协调，动态路由和协商 | 灵活、低门槛、适应性强 | 不可预测、可能无限对话 |

## 常见误区

- **误区 1**：Agent 越多越好 → Agent 越多，通信开销和协调复杂度越大。应该用最少数量满足需求
- **误区 2**：拆分 = 架构设计 → 拆分应该来自真实需求（token 超限、并行收益、失败隔离），不是来自「设计」
- **误区 3**：多 Agent 框架搭起来就是系统 → 框架搭起来是起点，真正的难点在 Agent 之间的通信设计、任务分配逻辑、失败传播处理

## 相关概念对比

- **vs [[concepts/Agent-Loop]]**：Agent-Loop 是单个 Agent 的推理循环，Multi-Agent 是多 Agent 的协作编排——后者通常由多个 Agent-Loop 组成
- **配合 [[concepts/Agent-Execution-Environment]]**：多 Agent 并发运行时，执行环境的隔离、资源调度、失败兜底是核心工程难点
- **配合 [[concepts/AI-Project-Differentiation]]**：「我用了多 Agent 协同」无信息量；「我在多 Agent 协同中遇到了 X 问题，用 Y 方案解决」才有差异化
