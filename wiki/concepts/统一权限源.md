# 统一权限源（Single Permission Source）

一种权限架构模式：全系统有且仅有一套权限规则，由操作系统内核强制执行，任何协议的文件访问均受同一套规则约束。

## 核心思想

不是通过同步机制保持多套权限副本一致，而是通过让所有协议共享内核 ACL 这一唯一权限执行器来彻底消除"不一致"的可能性。

## 为什么重要

> 一次 `setfacl` 调用的变更立即对所有协议可见，无延迟，无同步，不会漂移。相比自研权限表的方案，引入的代码量更少，可靠性更高，维护成本更低。

## 什么时候用

- 多协议文件访问共存（SMB + NFS + WebDAV + HTTP API）
- 需要权限变更"立即生效"的安全场景
- 需要权限控制的可靠性高于自定义灵活性的场景

## 什么时候不要用

- 权限规则极其复杂且无法映射到 POSIX ACL 模型（如基于内容的动态权限）
- 文件系统不支持 POSIX ACL（如某些嵌入式文件系统）
- 需要跨设备/跨集群的分布式权限控制（ACL 是本地文件系统的概念）

## 常见误区

- **误区**：权限源统一意味着所有协议的行为完全一致 → 实际上不同协议的认证方式和能力边界仍有差异，但权限判断结果完全一致
- **误区**：应用层校验就够了 → 应用层 bug 可能导致权限绕过，内核 ACL 是最后的防线

## 相关概念对比

- **vs [[concepts/统一身份源]]**：两者互补——身份决定"是谁"，权限决定"能做什么"。本架构中两者分别以 LDAP 和 POSIX ACL 实现
- **vs RBAC**：RBAC 是角色层面的抽象，统一权限源是底层执行层面的架构承诺。两者可共存（本架构中 admin/user 角色即 RBAC 层，POSIX ACL 是执行层）
