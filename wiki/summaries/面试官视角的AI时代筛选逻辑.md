# 面试官视角的 AI 时代筛选逻辑

**核心论点**：AI Coding 工具普及后，筛选优秀候选人的难度反而上升了——不是「候选人变强了好筛」，而是 AI 把实现能力这门过去的分水岭抹平了，所有人都能搭出 Agent demo，但面试真正在筛的「判断力」却被淹没了。

## 论证逻辑链

### 1. 「会用 AI」已从加分项变成默认项

> 当一个技能人人都会，它就不再是加分项，而是默认项。默认项的意思是——没有它你会被筛掉，有它你也不会因此被选中。

简历上普遍出现的 LangGraph / MCP / 多智能体，已经不是亮点而是标配。工具普及的速度远快于理解深度的积累。

### 2. 面试中的三个追问维度

作者将揭示判断力的追问分成三类：

**(1) 关于「为什么」** — 为什么拆成多个子 Agent，而不是一个 Agent 加工具调用？

大多数候选人的回答停留在「职责分离更清晰」这种教程层面的表述。真正有理解的人会说出具体取舍：
- 「规划链路和执行链路用同一套上下文会导致 token 太长且互相干扰，所以拆开」
- 「工具调用环节容易失败，单独拆成子 Agent 便于隔离重试，不影响主流程已积累的推理」

> 「为什么」是最难造假的维度。AI 能告诉你多 Agent 的好处是什么，但 AI 不能告诉你「你当时为什么这么拆」，因为这个「当时」只存在于你自己的经历里。

**(2) 关于边界** — Agent 什么情况下不该自主决策？

这个问题筛掉的人比想象中多。很多候选人会说「加了置信度阈值」「做了 fallback 机制」，但作者想要的答案是**「什么时候根本不该让 Agent 决策」**：
- 不可逆操作（下单、删除、转账、发邮件对外）→ Agent 不应自主执行，必须人工确认
- 低确定性判断（需要业务规则或领域知识）→ Agent 可以提建议但不能拍板
- Agent 不知道自己不知道 → 要靠规则把决策权拉回人工

**(3) 关于失败** — Agent 跑飞了，你是怎么定位和兜底的？

这是全文最核心的追问，将失败分成了两层：

| 失败层 | 现象 | 解决方案 |
|--------|------|---------|
| **逻辑层** | 推理循环失控、反复调用同一工具、自我对话死循环 | 给循环加状态约束、限制单轮工具调用次数、引入 critic 节点监督 |
| **执行环境层** | 沙箱不够隔离、多 Agent 环境互相污染、资源无限制导致 OOM 连锁影响其他任务 | 每个 Agent 跑在独立容器、超时自动回收、资源配额控制 |

> Agent 跑飞往往出事的地方，不是推理，而是执行环境——因为推理层的失败你能从日志看出来，执行环境层的失败往往是隐性的、累积的、甚至会连锁影响其他任务。

这也是为什么简历上写「每个 Agent 任务跑在独立容器里、超时自动回收」的信息量远大于「我用了多 Agent 协同架构」——前者说明你碰过真实问题。

### 3. AI 时代面试在筛的三层判断力

1. **需求判断**：能分清「这个需求里什么是真问题，什么是噪音」
2. **方案判断**：能在多个可行方案中选出适合当前场景的，并说清为什么
3. **边界判断**：能说清「这个方案在什么情况下会失效，失效了怎么办」

### 4. 给候选人的核心建议

1. 简历项目**少而深**，一个能扛住深度追问的项目胜过一堆 demo
2. 把「我用 AI 做了 X」换成「我判断 X 不该用 AI 做，因为……」
3. 面试前对每个项目问：为什么这么设计？最深坑是什么？重做怎么改？
4. 真实实习经历价值比往年都高——只有真实业务会逼你做取舍
5. 写 Agent 项目时多花一句话讲**执行环境**——跑在哪、怎么隔离、失败怎么兜底

> AI 能帮你把代码写出来，但面试官想看的是你这个人能不能扛事——代码只是你扛事的载体，不是扛事本身。
